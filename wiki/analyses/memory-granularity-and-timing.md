# 记忆的粒度与时机：原子事实 vs 叙事块、即时写入 vs 延迟总结

> 触发问题：[[memory-write-strategy|写入策略]] 解决了"该不该写"，[[memory-staleness-and-forgetting|遗忘机制]] 解决了"该不该留"，中间还有两个被忽略的轴——**写多大一块** 与 **什么时候写**。两者还彼此耦合：选错粒度，时机就救不回来；选错时机，粒度就配不齐。

## 结论

**粒度与时机不是各自独立的两个旋钮，而是一对必须协同设计的耦合参数。** 原子事实天生属于即时写入路径——快、便宜、易覆盖；叙事块天生属于延迟总结路径——必须看完全场才下得了笔。强行把它们错配（即时写叙事 / 延迟写原子）会同时损失信噪比与召回。一个健康的记忆系统应该是 **双轨并行 + 单一权威源**：原子事实做检索入口，叙事块做正文上下文，与 [[llm-wiki-pattern|LLM Wiki 模式]] 的 index → 正文二级架构同构。

## 一、粒度：原子事实 vs 叙事块

| 维度 | 原子事实 | 叙事块 |
|---|---|---|
| 形式 | "用户偏好 Go" | "用户在 X 项目里因 Y 性能瓶颈从 Python 迁到 Go，迁移后 Z 模块仍保留 Python" |
| 检索友好度 | 高（短、向量稳定、关键词命中） | 低（多主题混合、易被稀释） |
| 覆盖更新成本 | 低（直接 overwrite 一行） | 高（要么追加扩散，要么整块重写） |
| 误抽样风险 | 高（脱离 why，断章取义） | 低（自带边界条件） |
| 跨场景复用 | 强 | 弱（叙事自带语境绑定） |
| 易过时 | 中（标识符密度低） | 高（含具体决策、人名、版本） |
| 失败模式 | "记得偏好 Go，忘了边界条件" | "检索回来一大坨，但不知道哪句是答案" |

### 粒度选择三条判据

1. **是否会被多场景共用？** 共用度高 → 原子；共用度低 → 叙事。"偏好中文回复"是原子；"在这个支付项目里因合规要求只能用 Java"是叙事。
2. **why 是否非显然？** why 不写下来下次就推不出来 → 必须叙事；why 即结论 → 原子足够。
3. **是否易过时？** 含具体路径/版本/人名 → 倾向叙事（因为 [[memory-staleness-and-forgetting|陈旧记忆]] 一旦命中就需要边界条件帮 agent 判断"还成立吗"）；抽象偏好 → 原子（更新频率低）。

### 二级架构：原子做入口，叙事做正文

这与 [[llm-wiki-pattern|LLM Wiki 模式]] 的 index → 正文一致，与 [[context-governance|上下文治理]] 的"MEMORY.md 是索引，不是日记本"也一致：

- **检索层**：用原子事实做向量索引 + 关键词命中，召回快、覆盖广
- **上下文层**：命中后顺指针带回对应叙事块，把 why / 边界 / 例外注入 prompt
- **避免常见错误**：把叙事块当成检索单元（召回稀释）；把原子事实当成上下文（agent 拿到却用错）

## 二、时机：即时 vs 延迟 vs 触发式

| 时机 | 优势 | 失败模式 | 适配粒度 |
|---|---|---|---|
| **每轮即时写** | 响应快、不丢细节、写入失败可立即重试 | 噪音多、误判多、短期偏差被冻结 | 原子事实 |
| **Session 结束总结写** | 信噪比高、能综合全场、自动去重 | 丢中间细节、写入失败损失整个 session、延迟可见 | 叙事块 |
| **触发式**（用户明令 / 置信度阈值 / 关键决策节点） | 精准、可审计、误记率最低 | 召回低、依赖触发判定本身的可靠性 | 两者皆可，但偏叙事 |

### 三种时机各自的失败模式细看

- **即时写的"短期偏差冻结"**：用户当前会话用 Python，模型立刻写"用户偏好 Python"——其实只是这次任务用 Python。对应 [[memory-write-strategy|写入策略]] 的"稳定性 ≥ K 次复现"判据，本质上是用粒度 + 时机绕开短期偏差。
- **延迟总结的"写入失败放大"**：session 结束才写，crash / 上下文截断 / compact 失误任何一处出问题，整个 session 的沉淀清零。需要中间态 checkpoint 兜底。
- **触发式的"召回坑"**：依赖"模型自己识别该写"——而模型对自己置信度的判断本身就不可靠，参见 [[reflection-pattern|反思]] 必须独立通道的逻辑。

### 组合策略

实践里几乎不会单选一种，而是分层叠加：

1. 即时写：原子事实进入"待确认"层（低置信度、短 TTL）
2. Session 结束：扫一遍待确认层 → 高频/高确信的晋升为叙事块进主记忆
3. 触发式：用户明令或关键决策节点，直接以高置信度叙事块写入主记忆，跳过待确认

## 三、粒度与时机的耦合

| 组合 | 评价 | 说明 |
|---|---|---|
| 即时 + 原子 | 推荐 | 速度快、单条便宜、覆盖更新成本低；适配"待确认层" |
| 延迟 + 叙事 | 推荐 | 必须看完全场才能写出 why 和边界，session 结束是天然 commit 点 |
| 即时 + 叙事 | 反模式 | 每轮都试图凝结叙事，结果是写出半成品，后续要么不断追加要么反复重写 |
| 延迟 + 原子 | 浪费 | 原子事实根本不需要等到 session 结束才能定型，延迟只增加丢失风险 |

### 双轨并行如何避免重复写入

即时层与延迟层同时上线时，最大的风险是同一信息被记两次（原子层一条 + 叙事层一段），且两者还可能不一致。三条对策：

1. **单一权威源**：叙事块是权威，原子事实是叙事的"投影"——session 结束总结时，由叙事块**生成 / 覆盖**对应的原子条目，而不是各自独立写
2. **指针而非副本**：原子事实里只放叙事块的引用（类似 wikilink），不复制内容，避免 [[memory-staleness-and-forgetting|陈旧记忆]] 中"两份副本不同步"的二次伤害
3. **晋升而非追加**：即时写的待确认条目，在 session 结束被叙事块吸收后**应被回收**，不留双份

这三条本质上把 [[context-governance|上下文治理]] 的"索引 vs 正文"分层规则搬到了写入路径上。

## 与本 wiki 其他视角的呼应

- [[memory-write-strategy|写入策略]] 关心"什么值得写"——本页关心"既然要写，写多大、什么时候写"。两者是写入决策的不同切面。
- [[memory-staleness-and-forgetting|陈旧与遗忘]] 论证了易腐 vs 稳定要在写入侧分流；本页论证 **粒度也要在写入侧分流**——同一个写入瞬间需要做两次分类（粒度 + 易腐性）。
- [[prompt-chaining-in-long-context-era|长上下文时代下的提示词链]]：长 context 让"装得下叙事"成立，但运行时检索仍需要原子入口——这就是为什么二级粒度不会被长 context 取代。

## 一句话总结

> 粒度决定"记成什么形状"，时机决定"什么时候定型"。原子配即时、叙事配延迟，是因为它们在 **写入成本、覆盖代价、信噪要求** 三条曲线上恰好对齐；错配的代价不是慢一点，而是写出一堆半成品记忆，既不好查也不敢用。
