# 记忆的写入策略：信号与噪音的边界

> 触发问题：[[memory-management-pattern|记忆管理]] 关心"怎么存、怎么取"，但更前置的问题是——**什么值得被写进记忆**？写入策略才是长期记忆质量的瓶颈。

## 结论

**写入门槛比检索算法更决定记忆系统的长期信噪比。** 召回阶段做得再花哨，也救不回一个被无差别灌入的记忆库；反过来，一个克制的写入策略能让最朴素的检索都显得聪明。判断"该不该写"有三条可操作的硬标准，且"不该记的清单"比"该记什么"更能体现工程成熟度。

## 写入门槛：宽 vs 严的两难

| 写入策略 | 表现 | 失败模式 |
|---|---|---|
| 太宽（什么都记） | 记忆库膨胀，召回时高相关度结果被噪音稀释 | 退化成低质量 RAG，检索越多越糊 |
| 太严（只记显式） | 只剩用户明令保存的内容 | 退化成手工笔记，agent 失去"积累"能力 |
| 阈值化（带判据） | 写入数量级稳定、可审计 | 需要前置判断成本，但可摊销 |

[[context-governance|上下文治理]] 强调"治理得好的系统看起来有点吝啬"——这条原则同样适用于记忆写入：**吝啬的写入是优点，不是缺点**。

## 三条可量化的写入判据

不靠"感觉重要"，而是靠可检验的信号：

1. **未来 N 轮被引用概率** — 如果一条事实在接下来的 N 个会话里没有任何检索命中，应触发降权或淘汰；写入时若估计该概率显著低于阈值，直接不写。
2. **可推导性测试** — 能从代码、git log、文件系统当前状态读出来的事实，**不写**。记忆只承载"无法从世界状态恢复"的部分。
3. **用户偏好的稳定性** — 短期偏好（"这次用 Python"）不写，跨会话重复出现 ≥ K 次的偏好才写；区别于一次性指令。

这三条对应 [[harness-ten-principles|Harness 十原则]] 中"上下文是工作内存"的精神：**记忆是预算，不是垃圾桶**。

## 隐式写入 vs 显式写入

写入路径的"出处"决定了它的可信度，应当分离：

| 维度 | 显式写入 | 隐式写入 |
|---|---|---|
| 触发 | 用户说"记住 X" | 模型从对话推断"用户似乎偏好 X" |
| 置信度 | 高 | 低，且容易过拟合短期对话偏差 |
| 风险 | 误记的几率主要来自理解错误 | 误抽样、回声室效应、把噪音当信号 |
| 治理建议 | 直接写入主记忆 | 写入"待确认"层，N 次复现后才晋升 |

**实践模式**：两条路径都应带置信度标签（`confidence: high/medium/low`）和来源标签（`source: explicit/inferred`）。检索时高置信记忆优先；隐式记忆需要在使用前显示给用户做被动确认（"我注意到你似乎倾向 X，对吗？"）。这与 [[reflection-pattern|反思]] 的精神一致——隐式信号必须经过独立通道的复核，不能在同一 context 里既抽样又使用。

## 不该被记的清单（更重要）

成熟的写入策略往往用排除法定义。以下内容**不应进入长期记忆**：

| 类别 | 例子 | 为什么不写 |
|---|---|---|
| 代码可读出的事实 | 项目用 React、入口是 main.ts | 读文件即得，写入只增加噪音 |
| Git 可查的历史 | "上周修了登录 bug" | git log 是权威源，记忆是劣化副本 |
| Ephemeral 状态 | 当前 PR 编号、临时分支名 | 生命周期短于记忆衰减周期 |
| Conversation-scoped 中间产物 | 本次会话的草稿、推理链 | 属于 [[context-governance\|Session Memory]]，不该跨会话 |
| 可由工具实时回答的 | 当前时间、当前文件树 | 工具调用比记忆检索更新鲜 |
| 单次性指令 | "这次帮我用中文回答" | 没有跨会话延续性 |

这张表本质上在划分一条边界：**记忆只负责"无法从外部世界恢复且具有跨会话价值"的信息**。其余都是 [[context-governance|上下文治理]] 范畴的工作内存或工具调用范畴的查询，不该污染记忆层。

## 与本 wiki 其他视角的呼应

- [[llm-wiki-pattern|LLM Wiki 模式]] 本身就是一种带写入策略的长期记忆：源文件不可变（raw/），wiki/ 层只承载"综合后的、可跨会话复用的"知识——天然遵守"可推导性测试"。
- [[prompt-chaining-in-long-context-era|长上下文时代下的提示词链]] 论证了"窗口大不等于该塞满"；同构地，**记忆容量大不等于该写满**。
- [[knowledge-management|知识管理]] 视角下，写入策略 = 知识录入的 schema 与去重规则，缺了它整个 wiki 会熵增。

## 一句话总结

> 记忆系统的长期质量，不取决于你记得多努力，而取决于你**敢不敢不记**——能从代码、git 或工具实时读出的，永远不该进记忆；隐式推断出的，永远要带置信度并经过复核才晋升。**写入策略是记忆的免疫系统**。
