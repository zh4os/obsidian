# 记忆 vs 工作流状态：常被混淆的两种持久化

> 触发问题：当人们说"agent 需要记忆"时，真正需要的常常是 **显式状态机** 或 **工作流 state**。把流程性状态塞进语义记忆，是 agent 工程里最高频的误用之一。

## 结论

**「按相似度匹配」和「按位置/顺序访问」是两种本质不同的持久化路径，不能用一种实现假装另一种。** 语义记忆解决"我以前见过类似的吗"，工作流 state 解决"我现在走到第几步、还差什么"。混淆的代价不是性能问题，而是 **正确性问题**：状态会幽灵复现、被覆盖、丢一致性。判据简单——四问就能分流；架构清晰——短期 state 与长期 memory 必须分层，且默认 **不互通**。

## 判据：该走记忆 vs 该走 state

四个问题，任何一个偏向右列就走 state：

| 维度 | 该走 [[memory-management-pattern\|记忆]]（语义检索） | 该走 工作流 state（按位置访问） |
|---|---|---|
| 访问方式 | 按相似度匹配（"以前有没有类似的"） | 按 key/位置/顺序读取（"第 3 步的输出是什么"） |
| 是否被覆盖 | 极少；以追加为主 | 频繁覆盖，每步都在变 |
| 生命周期 | 跨会话，长于任务本身 | 与一次任务/会话对齐，任务结束即销毁 |
| 一致性要求 | 最终一致即可，召回少一条不致命 | 强一致 / 事务性，丢一步整条链就错 |
| 唯一性 | 模糊匹配可接受重复 | 必须唯一可寻址（todo #3 就是 #3） |
| 失败模式 | 噪音稀释、召回偏 | 状态错乱、并发冲突、链路断裂 |

简言之：**记忆是"可有可无的回忆"，state 是"少一项就崩的账本"。** 两者写错对方位置都会失效。

## 混淆的三种典型代价

| 误用 | 现象 | 根因 |
|---|---|---|
| 把 todo list 塞进记忆 | 检索时召回不全（漏掉 todo #4），跨 session 出现"幽灵 todo"（上次任务残留被相似度匹配回来） | todo 的访问模式是按编号枚举，不是按相似度；且生命周期与任务对齐 |
| 把用户长期偏好塞进 plan/state | 会话一结束偏好就丢，下次重新询问 | 偏好的生命周期跨会话，必须落到长期记忆层 |
| 把中间产物（草稿、推理链）塞进记忆 | 污染未来检索，越用越糊；新任务召回到旧任务的草稿 | 中间产物属于 [[context-governance\|Session Memory]]，没有跨会话价值 |
| 把工具实时可查的信息塞进记忆 | 记忆里的"当前分支"过时，agent 基于陈旧事实行动 | 工具调用比记忆新鲜，记忆是劣化副本 |

这与 [[memory-write-strategy|记忆写入策略]] 中的"不该被记的清单"互为镜像——那张表回答"什么不该写入记忆"，本文回答"那些不该写入记忆的，应该走哪里"。

## 双层架构：state 与 memory 默认不互通

```
┌──────────────────────── 短期：工作流 state ────────────────────────┐
│  plan / scratchpad / todo / 中间产物 / 工具调用账本                │
│  - 结构化（JSON、显式 schema）                                     │
│  - 按位置/key 访问                                                 │
│  - 生命周期 = 一次任务/会话                                        │
│  - 强一致、可枚举、可序列化                                        │
└────────────────────────────────┬───────────────────────────────────┘
                                 │  显式 promote（少数情况）
                                 ▼
┌──────────────────────── 长期：semantic memory ─────────────────────┐
│  用户偏好 / 跨会话事实 / 历史交互摘要                              │
│  - 向量化、按相似度检索                                            │
│  - 跨会话                                                          │
│  - 最终一致即可                                                    │
└────────────────────────────────────────────────────────────────────┘
```

**衔接点：何时该 promote（state → memory）？**

只在三种情况下把 state 内容升级到 memory：

1. **重复出现 ≥ K 次的稳定偏好** — 一次性指令（"这次用 Python"）不升级；多次会话都出现的偏好才升级。
2. **任务结束后产生的可复用结论** — 例如 [[query-workflow|Query 工作流]] 产出的 `analyses/` 页面，是被显式综合过、去过中间步骤的成品，不是原始 scratchpad。
3. **用户显式指令** — "记住 X"，但仍要按 [[memory-write-strategy|写入策略]] 做置信度标签。

**反过来不该升级的（默认全部）：** 进行中的 todo、未完成的 plan、推理草稿、工具返回的临时结果、当前分支号、本次会话的中间假设。这些一旦进入 memory，就成为未来检索的污染源。

## 为什么这条边界容易被打破

- 框架同质化诱导：很多 agent 框架（如 Google ADK 的 Session/State/MemoryService）名义上分了三层，但实际 API 都是 KV 写入，开发者很容易把所有东西都丢进同一个 store，分层只剩文档约定。
- LLM 自然语言泛化能力强：模型能从语义记忆里"猜"出 todo，给人一种"记忆能干 state 的活"的错觉——直到一次跨会话出现幽灵 todo。
- 长上下文的副作用：当窗口足够大时，"全塞进 prompt"看起来都能 work，掩盖了 state 与 memory 在语义上的分离需求。但正如 [[prompt-chaining-in-long-context-era|长上下文时代下的提示词链]] 所论：**装得下不等于该这么装**。

## 一句话总结

> 「需要记忆」的需求，一半其实是「需要 state」。判据是四个字：**位置 vs 相似度**——按编号读的走 state，按"像不像"读的走 memory。把 todo 塞进 memory 会有幽灵，把偏好塞进 state 会失忆；两层默认不互通，只在有限的三种情况下才允许显式 promote。
