# 记忆的遗忘与陈旧：为什么记忆系统需要主动失效机制

> 触发问题：[[memory-management-pattern|记忆管理]] 默认讨论的是"怎么记住"，但工程上更难的问题是"怎么知道某条记忆已经过期了"。文件被重命名、人离职、决策被推翻——三个月后记忆里还写着"X 文件包含 Y"，这条记忆就从资产变成了幻觉源。

## 结论

**写入记忆和删除/失效记忆同等重要，且后者长期被低估。** 记忆系统如果只有"沉淀"没有"过期"，几个月后会从助力变成噪声源。陈旧记忆比缺失记忆更危险——它带来虚假自信，并主动抑制 agent 重新核对的动机。主动失效不是优化项，是正确性前提。

## 一、记忆怎么知道自己过时？四种失效机制对比

| 机制 | 触发时机 | 成本 | 召回率 | 适用记忆类型 |
|---|---|---|---|---|
| **访问时校验（lazy verification）** | 用之前 grep / stat 一下事实是否仍成立 | 每次读放大延迟，但只对真正用到的记忆付费 | 高（用到才查，必查） | 含具体路径/函数名/版本号的"易腐"记忆 |
| **周期性 lint（active sweep）** | 定期全量扫描，类似 [[lint-workflow\|Lint 工作流]] | 一次性高、可离线，命中后果是把陈旧条目降权或删除 | 中（取决于扫描频率） | 跨页面交叉引用、wikilinks 完整性 |
| **TTL（time-to-live）** | 写入时打过期戳，到点自动失效 | 极低，但需要按记忆类型分桶设置寿命 | 低（盲目过期，可能误杀稳定事实） | 决策类、临时偏好、调试现场 |
| **写入时打"易腐"标签** | 写入瞬间标注脆性来源 | 写入侧 + 元数据 schema 成本 | 高（与 lazy verification 联动效果最佳） | 任何含具体路径/PR 编号/人名/版本号的记忆 |

实践组合一般是 **写入时分类（易腐 vs 稳定）→ 易腐走 lazy verification + 短 TTL → 稳定靠周期性 lint 兜底**。单用任何一种都会有漏网。

## 二、易腐 vs 稳定：写入侧就该分流

| 记忆形态 | 腐烂率 | 例子 | 推荐策略 |
|---|---|---|---|
| 含具体路径/函数名 | 高 | "鉴权逻辑在 `auth/handler.go:42`" | 易腐标签 + 访问时 grep 校验 |
| 含人名/职位 | 高 | "X 在负责支付组" | 短 TTL（30-90 天）+ 来源链接 |
| 含版本号/PR 号 | 中 | "v2.3 起改用 Postgres" | 写入即冻结（标记历史快照） |
| 抽象判据/原则 | 低 | "上下文是工作内存，不是越多越好" | 长寿命，靠 lint 检查与新源冲突 |
| 用户偏好 | 中 | "用户偏好中文回复" | 周期性确认，而非默认续期 |

判据：**记忆里有没有可以被外部世界悄悄改写的标识符？** 有就是易腐。

## 三、记忆 vs git history：边界与判据

git 已经是事实库——谁改了什么、什么时候、相对前一版差异是什么。记忆库再去重复存这些是浪费且危险（容易脱节）。但实践中两者经常重叠：决策、设计取舍、临时绕过的"为什么"，既会进 commit message 也会进记忆。

| 维度 | git history 擅长 | 记忆系统擅长 |
|---|---|---|
| **What / When / Who** | 是事实库 | 不要重复，只要引用 |
| **Why（决策动机）** | commit message 偶有，但分散、不可检索 | 记忆主场——综合视角、跨决策的脉络 |
| **被否决的方案** | 不会留下（除非 PR 评论） | 应记录，否则同样的弯路反复走 |
| **跨仓库 / 跨人的判据** | 无 | 主场 |
| **临时现场（调试中间态）** | 不该进 | 短 TTL 进，过期自动清 |

**强制边界的判据**：写记忆前问一句"这条信息 git blame / git log 能查到吗？" 如果能，记忆里只留指针（commit hash + 一句解释 why），不复制内容。这样事实演化时只需要 git 自己更新，记忆永远不会撒谎。

这条规则与 [[context-governance|上下文治理]] 的"MEMORY.md 是索引，不是日记本"是同一原则在不同尺度上的体现。

## 四、陈旧记忆的二次伤害

缺失记忆的代价是 **额外搜索一次**；陈旧记忆的代价是 **基于错误事实自信地推进**，且因为"已有记忆"而抑制了重新核对的动机。这是不对称伤害。

| 失败模式 | 缺失记忆 | 陈旧记忆 |
|---|---|---|
| Agent 行为 | 触发检索/询问 | 直接引用，跳过核对 |
| 错误传播 | 局部、易发现 | 全链路、难溯源 |
| 用户体验 | 慢一点 | 看起来很自信地胡说 |
| 修复成本 | 补一次 | 先得发现错了，再回溯所有引用过它的下游 |

**推论**：写入门槛不应低于删除门槛。一个不会主动遗忘的记忆系统，长期来看比没有记忆更糟。这与 [[reflection-pattern|反思]] 模式的精神一致——系统必须对自己产物保持怀疑，而不是只产不验。

## 五、对本 wiki 的具体含义

- 本 wiki 的 [[lint-workflow|Lint 工作流]] 已经覆盖"missing pages / orphan pages / 缺失交叉引用"，但未覆盖"陈旧事实"。补一条：含具体外部标识符（路径、PR 号、版本号）的内容，应额外打 `易腐` tag，定期校验。
- `wiki/sources/` 下的 source 摘要天然稳定（指向不可变的 `raw/`），不需要失效机制；但 `wiki/analyses/` 综合页一旦有新源进来，可能整体陈旧——应在新源 ingest 时同步检查相关 analysis 是否需要标记 stale。
- frontmatter 的 `updated` 字段目前是"最后修改时间"，可以再加一个 `verified` 字段（最后核对时间），让 lint 能据此挑出长期未核对的稳定页面。

## 一句话总结

> 记忆不是写下就完事的资产，而是会随真实世界漂移的债。**会写、会查、会忘** 三件事缺一不可——而"会忘"是大多数 agent 记忆系统当前最弱的一环，也是陈旧记忆变幻觉源的根因。
